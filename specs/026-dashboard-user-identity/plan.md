# Implementation Plan: Dashboard User Identity Display

**Branch**: `[026-dashboard-user-identity]` | **Date**: 2026-05-21 | **Spec**: [specs/026-dashboard-user-identity/spec.md](../spec.md)
**Input**: Feature specification from `/specs/026-dashboard-user-identity/spec.md`

## Summary

Display the logged-in user's name and email in the dashboard navigation menu (Sidebar), its mobile equivalent (MobileNav), and the TopBar user dropdown. Retrieve identity data from the existing `AuthContext` (stored in `localStorage` at `rushd_user`). Build a small display-name utility to gracefully compose `firstName` + `lastName` or fall back to `email` when name fields are missing. Ensure the UI remains responsive and visually consistent with existing dashboard styling.

# Phase 0: Research

**Status**: Complete ‚Äî no external research needed.

**Findings**:
- **Existing Auth System**: `useAuth` already exposes `user: UserProfile | null` and `isLoading: boolean`. User profile is persisted in `localStorage` under key `rushd_user`. No new data-fetching strategy is required.
- **Existing User Menu**: `TopBar.tsx` already contains a Radix-UI `DropdownMenu` + `Avatar` for the user, but values are hardcoded Arabic text (`ÿ£ÿ≠ŸÖÿØ ŸÖÿ≠ŸÖÿØ`, `ahmed@rushd.ai`). This should be replaced with dynamic values from `useAuth`.
- **Sidebar & MobileNav**: Neither currently displays user identity. Both are the canonical "dashboard menu" (navigation). A compact identity block should be added near the bottom of each, above the existing copyright footer.
- **Name Resolution**: `UserProfile` has optional `firstName?` and `lastName?`. The display-name utility must compose them and fall back to `email`.
- **No API Contracts Needed**: This feature consumes existing local state; it does not add new endpoints, props, or library interfaces.

# Phase 1: Design & Contracts

## Data Model: User Identity Display

This feature reuses the existing `UserProfile` type. No new entities are introduced.

### Reused Entity: `UserProfile`

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | `string` | Yes | Unique user identifier |
| `email` | `string` | Yes | User email address (fallback display) |
| `firstName` | `string` | No | First name for display composition |
| `lastName` | `string` | No | Last name for display composition |
| `companyName` | `string` | No | Not used for this feature |

### Display Name Resolution Logic

```
IF firstName AND lastName are present:
  displayName = "{firstName} {lastName}"
ELSE IF firstName is present:
  displayName = firstName
ELSE IF lastName is present:
  displayName = lastName
ELSE:
  displayName = email
```

**Fallback chain for avatar initials**:
```
IF firstName AND lastName are present:
  initials = firstName[0] + " " + lastName[0]
ELSE IF firstName is present:
  initials = firstName[0]
ELSE IF lastName is present:
  initials = lastName[0]
ELSE:
  initials = email[0]
```

## Contracts

**No external interfaces are added or modified for this feature.**

The feature reads from the existing `AuthContext` which is already the contract for accessing authenticated user data in the application. No new API endpoints, CLI commands, shared library interfaces, or page contracts are required.

## Quick Start

1. Open `/src/app/utils/getDisplayName.ts` ‚Äî create the display-name composition helper and its unit tests.
2. Update `/src/app/components/Sidebar.tsx` ‚Äî inject a `UserIdentity` block above the footer using `useAuth()`.
3. Update `/src/app/components/MobileNav.tsx` ‚Äî inject the same `UserIdentity` block above the footer using `useAuth()`.
4. Update `/src/app/components/TopBar.tsx` ‚Äî replace hardcoded values in the user `DropdownMenu` with `useAuth()` data.
5. Update `/src/app/utils/getDisplayName.ts` tests ‚Äî cover all fallback branches.
6. Run `npm run test` to verify unit tests, and `npm run dev` to verify in browser.
