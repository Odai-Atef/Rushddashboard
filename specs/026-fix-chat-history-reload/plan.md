# Implementation Plan: Fix Chat History Reload

**Branch**: `044-fix-chat-history-reload` | **Date**: 2026-06-13 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/026-fix-chat-history-reload/spec.md`

## Summary

Fix the chat history reload feature on `/dashboard/ai-analysis` so that clicking a history sidebar item renders the complete chat transcript with correct content, no empty messages, and visible follow-up Q&A. The fix involves: (1) adding a new `getSessionMessages(sessionId)` API method to fetch actual chat message records; (2) refactoring `sessionDetailToMessages()` to map `AnalysisMessage[]` → `StreamMessage[]` preserving order via `sequenceNo`; (3) updating the history detail DTO to use `description` instead of `content`; and (4) wiring the page-level `loadHistorySession()` to use the new method with fallback to corrected legacy behavior.

## Design Decisions

### DTO Field Correction
- **Change**: `AnalysisSessionDetail.insights[].content` → `description`
- **Rationale**: Matches actual `InsightResponseDto` from backend
- **Impact**: Fixes immediate empty message bug; enables fallback to work correctly

### New API Method
- **Addition**: `analysisService.getSessionMessages(sessionId: string)`
- **Endpoint**: `GET /api/v1/analysis/sessions/{sessionId}/messages`
- **Rationale**: Fetches actual chat transcript records instead of reconstructing from structured insights

### Refactored Message Conversion
- **Change**: `sessionDetailToMessages(detail: AnalysisSessionDetail)` → `messagesToStreamMessages(messages: AnalysisMessage[])`
- **Rationale**: Decouples message mapping from detail DTO; enables ordering by `sequenceNo`
- **Mapping rules**:
  - `AnalysisMessage.role` → `StreamMessage.role` (defensive: lowercase, unknown → `'assistant'`)
  - `AnalysisMessage.content` → `StreamMessage.content`
  - `isStreaming: false` for all historical messages
  - `AnalysisMessage.createdAt` → `StreamMessage.timestamp`
  - Sort by `sequenceNo` ascending

### Fallback Strategy
- **Primary**: Call `getSessionMessages(sessionId)`. On 200 → use messages.
- **Fallback**: On 404 / network error / endpoint unavailable → call `getRunDetail(runId)`, map insights using corrected `description` field.
- **Rationale**: Progressive deployment safety; aligns with spec FR-007.
