# Feature Data Model: Reset Password Page and Token Flow

**Date**: 2026-06-09
**Source**: Backend `ResetPasswordDto`, frontend auth patterns, and feature specification.

---

## Entities

### Entity: ResetToken

Represents the time-bound credential sent to the user via email and consumed by the reset-password page.

| Field | Type | Source | Description |
|-------|------|--------|-------------|
| `token` | `string` | URL query parameter | Raw UUID token extracted from the reset link (e.g., `?token=uuid-here`) |
| `isPresent` | `boolean` | Derived | True when a token string exists in the URL; used to decide whether to show the form or an error |

**Validation Rules**:
- Must be a non-empty string
- Must conform to UUID format (enforced by backend `@IsUUID()`)

**Lifecycle**:
1. Generated by backend during forgot-password flow
2. Sent to user via email
3. Consumed by frontend via URL query parameter
4. Submitted to backend in `POST /auth/reset-password`
5. Backend marks as used or leaves to expire

### Entity: PasswordResetForm

Represents the UI form state for collecting and validating the new password.

| Field | Type | Source | Description |
|-------|------|--------|-------------|
| `newPassword` | `string` | User input | The new password to set |
| `confirmPassword` | `string` | User input | Password confirmation for UX verification |
| `isSubmitting` | `boolean` | Derived | True while the API request is in flight |
| `status` | `'idle' \| 'success' \| 'error'` | Derived | Current submission status |
| `errorMessage` | `string` | Derived | User-facing error message for the current error state |

**Validation Rules** (client-side, enforced before submission):

| Rule | Field | Condition | Error Message (Arabic) |
|------|-------|-----------|------------------------|
| Required | `newPassword` | Non-empty | "كلمة المرور الجديدة مطلوبة" |
| Minimum length | `newPassword` | ≥ 8 characters | "يجب أن تكون كلمة المرور 8 أحرف على الأقل" |
| Uppercase letter | `newPassword` | Contains `[A-Z]` | "يجب أن تحتوي كلمة المرور على حرف كبير واحد على الأقل" |
| Lowercase letter | `newPassword` | Contains `[a-z]` | "يجب أن تحتوي كلمة المرور على حرف صغير واحد على الأقل" |
| Number | `newPassword` | Contains `[0-9]` | "يجب أن تحتوي كلمة المرور على رقم واحد على الأقل" |
| Special character | `newPassword` | Contains `[^A-Za-z0-9]` | "يجب أن تحتوي كلمة المرور على رمز خاص واحد على الأقل" |
| Confirmation match | `confirmPassword` | Exactly equals `newPassword` | "كلمتا المرور غير متطابقتين" |

**Note**: The backend `ResetPasswordDto` enforces the same rules (see `research.md`). The frontend validates client-side to provide immediate feedback and reduce unnecessary backend requests.

---

### Entity: PasswordStrengthRule

Represents an individual rule evaluated against the new password input, used for inline visual feedback.

| Field | Type | Description |
|-------|------|-------------|
| `label` | `string` | Human-readable description of the rule (e.g., "8+ characters") |
| `validator` | `(password: string) => boolean` | Function that returns true if the rule is satisfied |
| `isSatisfied` | `boolean` | Derived state for the current password value |

**Rules to Display**:
1. Minimum 8 characters
2. At least one uppercase letter
3. At least one lowercase letter
4. At least one number
5. At least one special character

## State Transitions

### ResetPasswordPage State Machine

```
[Initial Load]
  │
  ├─ token missing from URL ──→ [Error: Missing Token]
  │                                └── "الرابط غير صالح. اطلب رابطًا جديدًا"
  │
  └─ token present ──→ [Form State: Idle]
                          │
                          ├─ user types password ──→ [Real-time validation]
                          │                          │
                          │                          ├─ all rules pass ──→ [Submit enabled]
                          │                          └─ some rules fail ──→ [Submit disabled, inline errors]
                          │
                          ├─ user clicks submit ──→ [Submitting]
                          │                          │
                          │                          ├─ backend 200 ──→ [Success]
                          │                          │                    └── redirect to /auth/login after delay
                          │                          │
                          │                          └─ backend 400 ──→ [Error: Invalid Token]
                          │                                               └── show "الرمز غير صالح أو منتهي الصلاحية"
                          │                                                   + button to /auth/forgot-password
                          │
                          └─ network error ──→ [Error: Network]
                                                   └── show retry-friendly message
```

## Data Flow

1. **Token Extraction**
   - `ResetPasswordPage` mounts
   - Reads `token` from `URLSearchParams` (e.g., `/auth/reset-password?token=...`)
   - If missing, immediately renders error state

2. **Form Input**
   - User types into "New Password" and "Confirm Password" fields
   - Client-side validation runs on blur/change
   - Password strength indicator updates in real time

3. **Submission**
   - On submit, validate all client-side rules
   - If valid, call `authService.resetPassword(token, newPassword)`
   - Show loading spinner during API request

4. **Response Handling**
   - **Success**: Show success message, then `navigate('/auth/login')`
   - **Invalid token (400)**: Show unified error message with link to forgot-password
   - **Validation error (400)**: Show inline errors (e.g., password too weak — though client-side should prevent this)
   - **Network error**: Show generic retry-friendly message
